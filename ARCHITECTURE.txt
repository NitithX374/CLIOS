# C.L.I.O.S. (CLI-based OSPF Simulator) Architecture Overview

## 1. Introduction
C.L.I.O.S. is an interactive network simulation tool designed to model OSPF behaviors through a terminal interface. It integrates classical network logic with AI reasoning (LLM) and Retrieval-Augmented Generation (RAG) using RFC 2328 documentation.

## 2. Component Breakdown

### Main Application (chat.py)
 - Acts as the primary controller and user interface loop.
 - Manages mode switching ("topology" vs "ask").
 - Orchestrates the flow of data between the human user, the network state, and the AI agent.

### CLI & Visualization Engine (cli_engine.py)
 - Parses command strings into specific network operations (e.g., adding routers, configuring interfaces).
 - Houses the `_draw_topology` function, which uses Graphviz to generate real-time PNG diagrams (`topology_output.png`).
 - Performs real-time validation for area ID and area type mismatches between connected peers.

### Network State Management (network_state.py)
 - The internal database of the simulation.
 - Tracks the hierarchical structure of Routers -> Interfaces -> Links.
 - Stores metadata such as IP addresses, OSPF process status, Area IDs, Area Types (Stub, NSSA, etc.), and External Domain marking.

### AI Reasoning Engine (llm_engine.py)
 - Connects to the Typhoon/OpenAI-compatible API using a custom system prompt.
 - Interprets the exported topology context to answer technical questions about OSPF convergence and LSA behavior.

### RAG Knowledge Base (rag/)
 - retriever.py & embedder.py: Handle the semantic lookup of OSPF protocol knowledge from RFC 2328.
 - Uses a FAISS vector index (`faiss.index`) to provide context-accurate excerpts from the official OSPF documentation.

## 3. Data Flow
1. User enters a command in the CLI.
2. `cli_engine` updates the `network_state`.
3. If `show topology` is called, `cli_engine` triggers the Graphviz GUI.
4. If a question is asked in `ask mode`, `chat.py` exports the current state from `network_state` and sends it to `llm_engine`.
5. `llm_engine` retrieves relevant RFC context through `retriever` and generates a plain-text response (including emojis! 🤖).
